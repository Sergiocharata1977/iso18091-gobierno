## Agente A - Tokens y capas globales
**Puede ejecutarse en paralelo con:** Agente B, Agente C
**Depende de:** nada - es la primera ola

### Objetivo
Consolidar tokens visuales, semantica de color, radios, sombras y reglas globales para que el sistema use una base coherente en light y dark mode.

### Archivos a crear
- `src/design/tokens/ui-tokens.ts` - fuente central de tokens semanticos de color, radio, sombra y espaciado
- `src/design/tokens/ui-motion.ts` - duraciones, easing y reglas de motion/reduced motion

### Archivos a modificar
- `src/app/globals.css` - reemplazar overrides dispersos por una capa semantica mas limpia y predecible
- `src/config/theme-config.ts` - alinear definiciones con los nuevos tokens
- `src/components/ui/button.tsx` - consumir tokens y estados consistentes
- `src/components/ui/card.tsx` - alinear bordes, sombras y superficies
- `src/components/ui/badge.tsx` - normalizar badges semanticos

### Prompt completo para el agente
Trabaja en el repo `9001app-firebase`, stack Next.js 14 + React 18 + Tailwind v4 + componentes UI propios estilo shadcn. Lee como contexto `src/app/globals.css`, `src/config/theme-config.ts`, `src/components/ui/button.tsx`, `src/components/ui/card.tsx`, `src/components/ui/badge.tsx`. Implementa una capa de tokens semanticos para superficies, textos, bordes, estados, radios, sombras y motion. Reduce la dependencia de overrides globales para clases hardcodeadas y deja una estructura que permita que nuevas pantallas usen tokens en lugar de colores directos. No cambies navegacion, layout ni paginas concretas en esta tarea. Criterio de exito: existe una base visual unica, reutilizable y consistente entre light/dark sin romper las primitivas UI.

## Verificacion final

- El panel desktop y mobile comparten jerarquia, labels y estados activos coherentes.
- No quedan textos visibles con encoding roto en shell, auth y navegacion.
- Login, dashboard y shell se perciben parte del mismo producto.
- Los componentes base usan tokens semanticos en lugar de colores hardcodeados en la mayoria de los casos.
- Focus states, touch targets y contrastes cumplen una base de accesibilidad razonable.
- Dark mode no depende de tantos parches globales para clases especificas.
- Existe documentacion minima para copy, estados, rollout y adopcion del sistema visual.
